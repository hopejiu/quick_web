C1: 提取 SiteCoordinator — main.go 职责过载

main.go internal/hotkey/hotkey.go internal/webview/manager.go internal/scripts/loader.go

问题

  • main.go 包含热键注册逻辑 (registerEntryHotkey)、OnSave 回调中的 diff 算法、脚本注入调度 — 这些都是业务逻辑，不是单纯的编排
  • OnSave 回调的 diff 逻辑（遍历旧 entries → 注销热键 → 遍历新 entries → 注册热键 → 销毁已删窗口）约 30 行，硬编码在 main 中无法测试
  • 热键回调中混合了窗口显示、脚本加载、ExecJS 注入三个关注点
  • 违反 ADR-0001 声明的"main.go — 仅编排，不包含业务逻辑实现"

解决方案

提取 internal/coordinator/Service，将站点生命周期管理（热键注册/注销 + 窗口管理 + 脚本注入）封装为一个深模块。main.go 只负责创建服务并调用 coordinator.SyncEntries(old, new)。

C2: hotkey.Service — goroutine 泄漏

internal/hotkey/hotkey.go
Bug: Register() 启动 go func() { for range nhk.Keydown() { ... } }()，但 Unregister() 只调用 hk.Unregister()，不会 close channel，goroutine 永远阻塞。

问题

// Register 中的 goroutine
go func() {
    for range nhk.Keydown() {  // ← channel 永远不会 close
        callback()
    }
}()

// Unregister 只注销热键，不通知 goroutine 退出
func (s *Service) Unregister(keystr string) {
    hk.Unregister()  // ← goroutine 继续阻塞在 Keydown()
}

解决方案

用 context.WithCancel 控制 goroutine 生命周期：

type entry struct {
    hk   *hotkey.Hotkey
    stop context.CancelFunc
}

func (s *Service) Register(keystr string, callback func()) error {
    ctx, cancel := context.WithCancel(context.Background())
    // ...
    s.entries[keystr] = &entry{hk: nhk, stop: cancel}
    go func() {
        for {
            select {
            case <-ctx.Done():
                return
            case <-nhk.Keydown():
                callback()
            }
        }
    }()
}

func (s *Service) Unregister(keystr string) {
    if e, ok := s.entries[keystr]; ok {
        e.stop()          // ← 通知 goroutine 退出
        e.hk.Unregister()
        delete(s.entries, keystr)
    }
}

收益

  • 正确性：修复实际的 goroutine 泄漏
  • 可测试性：context 控制让 goroutine 生命周期可验证

C5: 脚本注入依赖 time.Sleep — 不可靠

main.go

问题

// 主窗口脚本注入
go func() {
    time.Sleep(600 * time.Millisecond)  // ← 硬编码延时
    scriptDir := sett.ScriptDir(initialSettings.Entries[0].ScriptDir)
    if siteScripts, _ := scripts.LoadFromDir(scriptDir); err == nil {
        for _, js := range siteScripts {
            win.ExecJS(js)  // ← 页面可能还没加载完
        }
    }
}()

600ms 是经验值，网络慢时页面可能未加载完，脚本执行失败。网络快时浪费 600ms。

解决方案

Wails v3 的 Window.HandleMessage("wails:runtime:ready") 已经注册了 runtime ready 事件。可以扩展为通用的 "页面就绪" 信号：

// 方案 A：使用 Wails 的 WindowEvents.Common.WebViewDidFinishNavigation
w.OnWindowEvent(events.Common.WebViewDidFinishNavigation, func(e *application.WindowEvent) {
    // 页面加载完成，注入脚本
})

// 方案 B：保留重试机制，但由 injector 检测注入结果
// ExecJS 返回 error 可以判断是否成功

收益

  • 可靠性：脚本在页面真正就绪后注入，不再依赖延时
  • 响应速度：快网络下立即注入，不再等待 600ms

C6: settings.Save — 锁语义不清

internal/settings/service.go

问题

func (s *Service) Save() error {
    s.mu.RLock()           // ← 读锁
    defer s.mu.RUnlock()
    return s.saveLocked()  // ← 但内部写磁盘
}

func (s *Service) saveLocked() error {
    data, _ := json.MarshalIndent(s.data, ...)  // 读 s.data — OK
    os.WriteFile(s.path, data, 0600)            // 写磁盘 — 但持的是读锁
}

json.MarshalIndent 只读 s.data，用 RLock 合理。os.WriteFile 写的是外部文件（非共享内存），不涉及 s.data。所以功能正确，但语义容易引起误解。

解决方案

重命名为 SaveSnapshot() 或在注释中明确说明 RLock 的合理性：锁只保护 s.data 的读取，磁盘写入不需要锁保护。

收益

  • 可读性：消除代码审查时的疑问
